# Reports Check

[![CI](https://github.com/Olgasn/reports-check/actions/workflows/ci.yml/badge.svg)](https://github.com/Olgasn/reports-check/actions/workflows/ci.yml)

Программный комплекс для автоматической проверки отчётов по лабораторным работам студентов с использованием больших языковых моделей (LLM).

## Назначение

Система предназначена для преподавателей и обеспечивает выполнение следующих задач:

- управление курсами, лабораторными работами, группами и студентами;
- импорт списков студентов из CSV-файла в формате выгрузки системы Moodle;
- загрузку задания лабораторной работы и архива студенческих отчётов либо отдельного файла;
- проверку каждого отчёта одной или несколькими языковыми моделями, выполняемую параллельно;
- формирование по каждому отчёту структурированного результата, включающего оценку, рецензию, перечень достоинств и перечень недостатков;
- формирование сводной рецензии агрегирующей моделью при проверке несколькими моделями;
- учёт результатов предыдущей проверки при повторной проверке отчёта того же студента;
- выявление попыток внедрения инструкций (prompt-инъекций) в текст студенческого отчёта и отображение соответствующего предупреждения в интерфейсе;
- отслеживание хода проверки в режиме реального времени.

Пользовательский интерфейс и все запросы к языковым моделям сформированы на русском языке.

## Архитектура и технологический стек

Комплекс реализован на платформе .NET 10 и построен в соответствии с принципами чистой архитектуры (Clean Architecture). Решение разделено на четыре проекта, объединяемые расширениями внедрения зависимостей `AddApplication` и `AddInfrastructure`. Файл решения — `ReportsCheck.slnx`.

| Слой | Назначение | Основные технологии |
|------|------------|---------------------|
| `ReportsCheck.Domain` | Доменные сущности, перечисления, интерфейсы репозиториев и единицы работы, константы оценки риска инъекций. Не имеет внешних зависимостей. | — |
| `ReportsCheck.Application` | Запросы, обработчики и валидаторы в стиле CQRS, объекты передачи данных и их отображение, конвейер проверки отчётов, сборка промптов, служба анализа инъекций. | MediatR, FluentValidation |
| `ReportsCheck.Infrastructure` | Доступ к данным, обработчики языковых моделей, разбор файлов, уведомления, фоновая очередь и обработчик заданий. | EF Core 10, SQLite, Semantic Kernel |
| `ReportsCheck.Web` | Веб-приложение и точка входа, компоненты пользовательского интерфейса, шаблоны промптов. | Blazor Server, MudBlazor 9, Serilog |

Пользовательский интерфейс реализован средствами Blazor Server с интерактивным серверным режимом отображения компонентов и библиотекой компонентов MudBlazor. Приложение не содержит обособленного программного интерфейса (REST API) и контроллеров: компоненты обращаются к прикладному слою непосредственно через посредник MediatR.

### Структура взаимодействия с данными

Доступ к данным осуществляется через обобщённый репозиторий и единицу работы (Unit of Work). Контекст базы данных применяет конфигурации сущностей, обнаруженные в сборке. Схема базы данных управляется механизмом миграций EF Core и не синхронизируется автоматически по модели; миграции применяются при запуске приложения.

### Конвейер проверки отчётов

Запрос на проверку формируется на странице загрузки и помещается в фоновую очередь, после чего управление немедленно возвращается пользователю. Обработка заданий выполняется фоновой службой в отдельной области внедрения зависимостей. Ход и результаты проверки передаются в интерфейс через службу уведомлений.

Стратегия проверки определяется числом выбранных моделей:

- при выборе одной модели она применяется ко всем отчётам параллельно;
- при выборе двух и более моделей все модели, кроме последней, применяются к каждому отчёту, после чего последняя модель выступает в роли агрегатора и формирует итоговую сводную рецензию на основании результатов остальных моделей.

Для каждого отчёта выполняется идентификация студента, детерминированный анализ текста на наличие инъекций, сборка промпта, обращение к языковой модели с повторными попытками при ошибках, извлечение и проверка структурированного результата, а также объединение полей безопасности.

## Поддерживаемые интерфейсы моделей

| Интерфейс | Описание |
|-----------|----------|
| `OpenAi`  | Любой провайдер с программным интерфейсом, совместимым с OpenAI. Требуются адрес провайдера и ключ доступа. При обращении к провайдеру передаются заголовки `X-Title` и `HTTP-Referer`, что позволяет идентифицировать запросы в панели управления провайдера. |
| `Ollama`  | Локальные модели, исполняемые средствами Ollama. Ключ доступа и провайдер не требуются. |

Связь с языковыми моделями обеспечивается посредством Semantic Kernel; выбор обработчика выполняется по значению интерфейса модели.

## Защита от внедрения инструкций

Система реализует многоуровневую защиту от попыток студентов изменить поведение языковой модели посредством внедрения инструкций в текст отчёта.

### Детерминированный анализ

Перед обращением к языковой модели служба `PromptInjectionService` сканирует текст отчёта по набору регулярных выражений и присваивает уровень риска:

| Уровень риска | Описание |
|---------------|----------|
| `none`        | Признаки внедрения инструкций не обнаружены. |
| `low`         | Обнаружены косвенные признаки. |
| `medium`      | Зафиксированы попытки манипуляции форматом вывода или служебными маркерами. |
| `high`        | Выявлены прямые попытки изменить роль модели, навязать оценку либо отменить проверку. |

Результат анализа, включающий уровень риска, перечень сработавших индикаторов и фрагменты подозрительного текста, передаётся в промпт через подстановочный маркер `@SECURITY_CONTEXT` в формате JSON. Это позволяет языковой модели учитывать контекст безопасности при формировании рецензии.

### Инструкции в промпте

Все шаблоны промптов содержат блок правил безопасности, предписывающих модели:

- рассматривать содержимое отчёта как ненадёжные данные, а не как инструкции;
- не выполнять команды, обнаруженные внутри отчёта;
- фиксировать сведения о выявленных нарушениях в структурированном ответе.

Текст студенческого отчёта отделяется от инструкций маркером `@SPLIT` и заключается в теги вида `<UNTRUSTED_...>`, что исключает его интерпретацию как части системных указаний. Служебные маркеры, встретившиеся внутри ненадёжного содержимого, обезвреживаются.

### Проверка ответа модели

После получения ответа система проверяет, что текст рецензии не содержит признаков выполнения внедрённых инструкций (например, формулировок о пропуске проверки по просьбе студента). При обнаружении подобных маркеров результат отклоняется, а проверка соответствующего отчёта считается неуспешной.

### Поля безопасности в результате проверки

Каждый результат проверки содержит следующие поля:

| Поле | Тип | Описание |
|------|-----|----------|
| `promptInjectionDetected` | логический | Признак обнаруженного внедрения инструкций. |
| `promptInjectionRisk` | строковый | Уровень риска (`none`, `low`, `medium`, `high`). |
| `promptInjectionFragments` | строковый | Подозрительные фрагменты текста. |
| `securityComment` | строковый | Комментарий о характере нарушения. |

Поля, сообщённые моделью, объединяются с результатами детерминированного анализа: выбирается более высокий уровень риска, а фрагменты текста объединяются без повторов. При обнаружении внедрения интерфейс отображает предупреждение с указанием уровня риска, комментария и извлечённых фрагментов.

## Форматы входных данных

### Задание лабораторной работы

Задание загружается файлом в формате `.pdf` или `.docx`; текст извлекается автоматически. Файлы иных форматов читаются как обычный текст в кодировке UTF-8.

### Архив отчётов студентов

Архив в формате ZIP, соответствующий выгрузке системы Moodle. Внутри архива располагаются папки студентов; имя папки должно соответствовать шаблону:

```
Фамилия Имя Отчество_<прочее>
```

Из каждой папки извлекается первый файл с расширением `.pdf`, `.docx` или `.doc`. Архив обрабатывается в оперативной памяти без распаковки на диск.

### Отдельный отчёт

Один файл в формате `.pdf`, `.docx` или `.txt`. Студент выбирается в интерфейсе либо определяется из имени файла по тому же шаблону `Фамилия Имя Отчество_<прочее>`.

### Импорт студентов из CSV

CSV-файл в формате Moodle (кодировка UTF-8 или Windows-1251, разделитель `,` или `;`). Обязательными являются следующие столбцы:

| Столбец | Альтернативное название |
|---------|-------------------------|
| `Имя`   | `firstname`             |
| `Фамилия` | `lastname`            |
| `Группы` | `groups`               |

Столбец `Отчество` (`middlename`) необязателен; при его отсутствии подставляется значение `-`. При совпадении записи по сочетанию имени, фамилии, отчества и группы строка пропускается. Группы, отсутствующие в базе данных, создаются автоматически.

## Шаблоны промптов

Шаблоны хранятся в каталоге `src/ReportsCheck.Web/Prompts/` и загружаются однократно при запуске приложения. Изменение файлов требует перезапуска приложения.

| Файл | Назначение |
|------|------------|
| `prompt.template` | Основной шаблон проверки одной моделью. |
| `prompt-prev.template` | Шаблон повторной проверки с учётом предыдущего результата. |
| `prompt-multiple.template` | Шаблон сводной рецензии при проверке несколькими моделями. |

Подстановочные маркеры в шаблонах:

| Маркер | Описание |
|--------|----------|
| `@PROMPT_TEXT`, `@PROMPT` | Пользовательские инструкции по проверке. |
| `@LAB_TASK` | Текст задания лабораторной работы. |
| `@STUDENT_ANSWER` | Текст отчёта студента (ненадёжные данные). |
| `@SECURITY_CONTEXT` | Объект JSON с результатами детерминированного анализа на внедрение инструкций. |
| `@MODELS_CHECK_RESULT` | Результаты проверки другими моделями (только в `prompt-multiple.template`). |
| `@PREV_*` | Данные предыдущей проверки (только в `prompt-prev.template`). |
| `@SPLIT` | Маркер границы между инструкциями и ненадёжными данными. |

Ответ модели обязан содержать блок JSON, заключённый в теги `<JSON>...</JSON>`; это требование зафиксировано во всех шаблонах и контролируется при разборе ответа.

## Конфигурация

Параметры приложения задаются в файле `src/ReportsCheck.Web/appsettings.json`.

| Параметр | Описание |
|----------|----------|
| `ConnectionStrings:Default` | Строка подключения к базе данных SQLite. Значение по умолчанию — `Data Source=reports-check.sqlite`. При отсутствии строки подключения используется значение ключа `DATABASE`. |
| `Llm:OllamaEndpoint` | Адрес локального сервиса Ollama. Значение по умолчанию — `http://localhost:11434`. |

Журналирование выполняется средствами Serilog: записи выводятся в консоль и в файлы `logs/reports-check-*.txt` с ежедневной ротацией.

## Требования и запуск

Для сборки и запуска требуется пакет средств разработки .NET SDK версии 10.

Команды выполняются из корневого каталога репозитория.

Сборка решения:

```bash
dotnet build ReportsCheck.slnx
```

Запуск веб-приложения:

```bash
dotnet run --project src/ReportsCheck.Web
```

Адреса приложения по умолчанию:

- `http://localhost:5034`
- `https://localhost:7147`

Миграции базы данных применяются автоматически при запуске приложения, поэтому отдельных действий для создания или обновления локальной базы данных, как правило, не требуется.

## Миграции базы данных

Изменение схемы базы данных требует создания новой миграции. Добавление миграции:

```bash
dotnet ef migrations add <Имя> --project src/ReportsCheck.Infrastructure --startup-project src/ReportsCheck.Web
```

Применение миграций вручную:

```bash
dotnet ef database update --project src/ReportsCheck.Infrastructure --startup-project src/ReportsCheck.Web
```

## Тестирование

Модульные тесты расположены в проекте `tests/ReportsCheck.UnitTests` и используют платформу xUnit совместно с библиотеками FluentAssertions и NSubstitute. Тесты, обращающиеся к EF Core, выполняются с провайдером SQLite.

Запуск всех тестов:

```bash
dotnet test
```

Запуск отдельного набора тестов:

```bash
dotnet test --filter "FullyQualifiedName~PromptInjectionServiceTests"
```

## Известные ограничения

- Система авторизации отсутствует: все страницы и операции доступны без аутентификации.
- Ключи доступа к провайдерам хранятся в базе данных без шифрования.
- При включённой повторной проверке используется последняя проверка студента без ограничения конкретной лабораторной работой.
- Студенты идентифицируются глобально по сочетанию фамилии, имени и отчества; два студента с одинаковыми фамилией, именем и отчеством в разных группах считаются одной записью.
- Конвейер проверки выполняется асинхронно: запрос на проверку завершается до начала обращений к языковым моделям, а сведения об ошибках поступают только через службу уведомлений и журнал.
- Корректная работа конвейера опирается на соблюдение соглашения Moodle об именовании файлов и папок вида `Фамилия Имя Отчество_<прочее>`.
- Детерминированный анализ на внедрение инструкций основан на фиксированном наборе регулярных выражений и не охватывает все возможные формы атак; окончательное решение о соответствии отчёта требованиям принимается языковой моделью.
- Уведомления о ходе проверки рассылаются всем подключённым клиентам без разграничения по пользователям; фильтрация по лабораторной работе выполняется на стороне страницы.
