# Reports Check

[![CI](https://github.com/Olgasn/reports-check/actions/workflows/ci.yml/badge.svg)](https://github.com/Olgasn/reports-check/actions/workflows/ci.yml)

Программный комплекс для автоматической проверки отчётов по лабораторным работам студентов с использованием языковых моделей (LLM).

## Назначение

Система предназначена для преподавателей и позволяет:

- управлять курсами, лабораторными работами, группами и студентами;
- импортировать списки студентов из CSV-файла (формат выгрузки Moodle);
- загружать задание лабораторной работы и архив отчётов студентов;
- выполнять проверку одной или несколькими LLM-моделями параллельно;
- получать по каждому отчёту оценку, рецензию, перечень достоинств и недостатков;
- при проверке несколькими моделями — формировать сводную рецензию агрегирующей моделью;
- учитывать результаты предыдущей проверки при повторной проверке отчёта;
- выявлять попытки prompt-инъекции в тексте студенческого отчёта и отображать предупреждение в интерфейсе;
- отслеживать ход проверки в реальном времени через WebSocket-уведомления.

## Архитектура

| Компонент | Технологии |
|-----------|------------|
| Клиент    | React 19, Vite, TypeScript, MUI, React Query, Redux Toolkit |
| Сервер    | NestJS 11, TypeORM, SQLite, Swagger, Socket.IO |

- База данных: SQLite, путь задаётся переменной окружения `DATABASE`.
- REST API доступен по префиксу `/api`.
- Документация API (Swagger): `http://localhost:3000/docs`.
- WebSocket-уведомления транслируются всем подключённым клиентам.

## Поддерживаемые интерфейсы моделей

| Интерфейс | Описание |
|-----------|----------|
| `OpenAi`  | Любой провайдер с OpenAI-совместимым API. Требуется URL провайдера и API-ключ. При обращении к API автоматически передаются заголовки `X-Title: Reports_Check` и `HTTP-Referer`, что позволяет идентифицировать запросы в дашборде провайдера. |
| `Ollama`  | Локальные модели через Ollama. Ключ и провайдер не требуются. |

## Защита от prompt-инъекций

Система реализует двухуровневую защиту от попыток студентов манипулировать поведением языковой модели посредством внедрения инструкций в текст отчёта.

### Детерминированный анализ

Перед отправкой запроса в LLM сервис `PromptInjectionService` сканирует текст отчёта по набору регулярных выражений и присваивает уровень риска:

| Уровень риска | Описание |
|---------------|----------|
| `none`        | Признаки инъекции не обнаружены |
| `low`         | Обнаружены косвенные признаки |
| `medium`      | Зафиксированы попытки манипуляции форматом вывода или служебными тегами |
| `high`        | Выявлены прямые попытки изменить роль модели, выставить оценку или пропустить проверку |

Результат анализа — уровень риска, список сработавших индикаторов и фрагменты подозрительного текста — передаётся в промпт через плейсхолдер `@SECURITY_CONTEXT`. Это позволяет языковой модели учитывать контекст безопасности при формировании рецензии.

### Инструкции в промпте

Все три шаблона промптов содержат блок правил безопасности, предписывающих модели:

- рассматривать содержимое отчёта как ненадёжные данные, а не как инструкции;
- не следовать командам, обнаруженным внутри отчёта;
- фиксировать сведения о выявленных нарушениях в структурированном JSON-ответе.

Текст студенческого отчёта отделён от инструкций маркером `@SPLIT`, что исключает его интерпретацию как часть системного промпта.

### Проверка ответа модели

После получения ответа сервер проверяет, что рецензия не содержит признаков выполнения инструкций инъекции (например, фраз вида «проверка пропущена по просьбе студента»). При обнаружении подобных маркеров результат отклоняется.

### Поля безопасности в результате проверки

Каждый результат проверки содержит следующие поля:

| Поле | Тип | Описание |
|------|-----|----------|
| `promptInjectionDetected` | `boolean` | Признак обнаруженной инъекции |
| `promptInjectionRisk` | `string` | Уровень риска (`none` / `low` / `medium` / `high`) |
| `promptInjectionFragments` | `string` | Подозрительные фрагменты текста |
| `securityComment` | `string` | Комментарий о характере нарушения |

При обнаружении инъекции интерфейс отображает предупреждение с уровнем риска, комментарием и извлечёнными фрагментами.

## Форматы входных данных

### Задание лабораторной работы

Задание загружается файлом в формате `.pdf` или `.docx`; текст извлекается автоматически. Файлы иных форматов читаются как обычный текст в кодировке UTF-8.

### Архив отчётов студентов

ZIP-архив в формате экспорта Moodle. Внутри архива располагаются папки студентов; имя папки должно соответствовать шаблону:

```
Фамилия Имя Отчество_<прочее>
```

Из каждой папки извлекается первый файл с расширением `.pdf`, `.docx` или `.doc`.

### Одиночный отчёт

Один файл `.pdf`, `.docx` или `.txt`. Студент выбирается в интерфейсе либо определяется из имени файла по тому же шаблону `Фамилия Имя Отчество_<прочее>`.

### Импорт студентов из CSV

CSV-файл формата Moodle (кодировка UTF-8 или Windows-1251, разделитель `,` или `;`). Обязательные столбцы:

| Столбец | Альтернативное название |
|---------|------------------------|
| `Имя`   | `firstname`            |
| `Фамилия` | `lastname`           |
| `Группы` | `groups`              |

Столбец `Отчество` (`middlename`) необязателен; при его отсутствии подставляется значение `-`. При совпадении записи (имя + фамилия + отчество + группа) строка пропускается. Группы, не существующие в базе данных, создаются автоматически.

## Шаблоны промптов

Шаблоны хранятся в каталоге `server/prompts/` и загружаются при запуске сервера. Изменение файлов требует перезапуска сервера.

| Файл | Назначение |
|------|------------|
| `prompt.template` | Основной шаблон проверки одной моделью |
| `prompt-prev.template` | Шаблон повторной проверки с учётом предыдущего результата |
| `prompt-multiple.template` | Шаблон сводной рецензии при проверке несколькими моделями |

Плейсхолдеры в шаблонах:

| Плейсхолдер | Описание |
|-------------|----------|
| `@PROMPT_TEXT` | Пользовательские инструкции по проверке |
| `@LAB_TASK` | Текст задания лабораторной работы |
| `@STUDENT_ANSWER` | Текст отчёта студента (ненадёжные данные) |
| `@SECURITY_CONTEXT` | JSON-объект с результатами детерминированного сканирования на инъекции |
| `@MODELS_CHECK_RESULT` | Результаты проверки другими моделями (только в `prompt-multiple.template`) |
| `@PREV_*` | Данные предыдущей проверки (только в `prompt-prev.template`) |
| `@SPLIT` | Маркер границы между инструкциями и ненадёжными данными |

Ответ модели должен содержать JSON-блок в тегах `<JSON>...</JSON>` — это требование зафиксировано во всех шаблонах и проверяется на сервере.

## Быстрый запуск (Windows)

Первый запуск — установка зависимостей:

```bat
install.bat
```

Последующие запуски:

```bat
start.bat
```

Скрипт запускает сервер, ожидает готовности порта 3000 и затем запускает клиент.

Адреса по умолчанию:

- frontend: `http://localhost:5000`
- backend: `http://localhost:3000`
- Swagger: `http://localhost:3000/docs`

## Ручной запуск

Требования: Node.js 18+, npm 9+.

Установка зависимостей:

```bash
npm install          # корень (только wait-port)
cd client && npm install
cd ../server && npm install
```

Запуск сервера (PowerShell):

```powershell
$env:DATABASE = "./db/dev.sqlite"
npm run start:dev
```

Запуск клиента (в отдельном терминале):

```powershell
$env:VITE_API_URL = "http://localhost:3000/api"
$env:VITE_WS_URL  = "http://localhost:3000"
npm run dev
```

## Запуск через Docker

```bash
docker compose up --build -d
```

Адреса:

- frontend: `http://localhost:8080`
- backend / Swagger: `http://localhost:3000/docs`

Постоянные тома (объявлены в `docker-compose.yml`):

- `./server/db:/app/db` — база данных
- `./server/prompts:/app/prompts` — шаблоны промптов

## Известные ограничения

- Отсутствует система авторизации: все эндпоинты API доступны без аутентификации.
- API-ключи хранятся в базе данных без шифрования.
- При включённой опции `checkPrev` используется последняя проверка студента без фильтрации по конкретной лабораторной работе.
- Студенты идентифицируются глобально по сочетанию «фамилия + имя + отчество»; два студента с одинаковыми ФИО в разных группах будут считаться одной записью.
- Схема базы данных управляется TypeORM в режиме `synchronize: true` (без миграций). Переименование или удаление полей сущностей приводит к потере данных в соответствующих столбцах.
- Детерминированный анализ на prompt-инъекции основан на фиксированном наборе регулярных выражений и не охватывает все возможные формы атак. Финальное решение о соответствии отчёта требованиям принимается языковой моделью.
- WebSocket-уведомления транслируются всем подключённым клиентам без разграничения по пользователям.
