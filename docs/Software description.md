## Архитектура программного комплекса

Приложение построено по клиент-серверной архитектуре. Серверная часть обрабатывает
запросы пользователя; для каждой предметной сущности выделен отдельный набор маршрутов,
обеспечивающих выполнение операций над ней. Серверная часть реализована на платформе
Node.js с использованием фреймворка NestJS, клиентская часть — в виде веб-приложения на
библиотеке React.

Для работы с сущностью «Ключ» предусмотрены следующие действия:

- создание ключа;
- редактирование ключа;
- удаление ключа;
- получение всех ключей;
- получение ключа по идентификатору.

Для работы с сущностью «Провайдер» предусмотрены следующие действия:

- создание провайдера;
- редактирование провайдера;
- удаление провайдера;
- получение всех провайдеров;
- получение провайдера по идентификатору.

Для работы с сущностью «Модель» предусмотрены следующие действия:

- создание модели;
- редактирование модели;
- удаление модели;
- получение всех моделей;
- получение модели по идентификатору.

Для работы с сущностью «Курс» предусмотрены следующие действия:

- создание курса;
- редактирование курса;
- удаление курса;
- получение всех курсов;
- получение курса по идентификатору;
- поиск курса по названию;
- получение лабораторных работ курса.

Для работы с сущностью «Промпт» предусмотрены создание и редактирование содержимого,
привязанного к конкретному курсу.

Для работы с сущностью «Лабораторная работа» предусмотрены следующие действия:

- создание лабораторной работы;
- редактирование лабораторной работы;
- удаление лабораторной работы по идентификатору.

Для работы с сущностью «Группа» предусмотрены следующие действия:

- создание группы;
- редактирование группы;
- удаление группы;
- получение списка групп;
- получение группы по идентификатору.

Для работы с сущностью «Студент» предусмотрены следующие действия:

- создание студента;
- редактирование студента;
- удаление студента;
- получение студентов по идентификатору группы;
- получение студента по идентификатору.

Для выполнения проверок отчётов по лабораторным работам предусмотрены следующие действия:

- разбор файла с отчётами — маршрут, который выполняет разбор архива с отчётами студентов
  и формирует на выходе список студентов, содержащихся в архиве;
- выполнение проверки — маршрут, который запускает проверку отчётов по лабораторным работам.

Для работы с результатами проверок предусмотрены следующие действия:

- получение проверок по идентификатору лабораторной работы;
- получение проверок по списку их идентификаторов.

Клиентская часть представляет собой веб-приложение, с которым пользователь взаимодействует
посредством браузера. Для удобства работы пользователя содержимое веб-страниц приложения
проработано в соответствии с выполняемыми функциями.

Для управления ключевыми параметрами системы предусмотрена отдельная страница настроек.
На ней пользователь получает возможность конфигурировать список доступных больших языковых
моделей, а также провайдеров, через которых осуществляется доступ к этим моделям. Кроме того,
на этой же странице выполняется управление ключами доступа (API-ключами).

Функции работы с курсами реализованы на странице «Курсы». На ней отображается полный
список созданных пользователем курсов. Для каждого отдельного курса предусмотрена своя
страница, на которой пользователь может просмотреть список всех привязанных к курсу
лабораторных работ, а также промпт, используемый при проверке лабораторных работ данного
курса.

Основной процесс выполнения проверок отчётов происходит на странице проверки. На ней
пользователь выбирает одну или несколько моделей для анализа, а затем загружает файл,
представляющий собой архив с отчётами студентов. Интерфейс страницы позволяет запускать
проверку и отслеживать её ход в реальном времени.

Для анализа и контроля уже выполненных проверок предусмотрена страница результатов. На ней
пользователь может просматривать результаты ранее выполненных проверок.

## Структура и основные алгоритмы

### Функциональные возможности и последовательность обработки информации

Основной целью программного комплекса является автоматизация процесса проверки отчётов
студентов по лабораторным работам с применением больших языковых моделей. Современные
языковые модели способны анализировать текст и выявлять в нём ошибки и недочёты.

Следует учитывать, что большие языковые модели принимают данные исключительно в текстовом
формате, поэтому перед проверкой отчёт студента по лабораторной работе должен быть преобразован
в текст. Для проверки отчёта формируется специальный запрос определённого формата, который
содержит разделы с заданием и отчётом студента.

После передачи запроса модель выполняет проверку отчёта студента и возвращает ответ,
содержащий следующие сведения:

- оценка отчёта студента по десятибалльной шкале;
- положительные стороны работы студента;
- недостатки и моменты, требующие доработки;
- краткий отзыв на работу студента.


Затруднения могут возникнуть, если отчёт содержит графический материал, поскольку модели
не анализируют содержимое изображений и не соотносят его с представленным текстом. Тем не
менее такие работы также могут быть проверены: при проверке подобного отчёта теряется лишь
часть информации.

Сервисы, предоставляющие доступ к большим языковым моделям, могут быть временно недоступны
по различным причинам или иметь ограничения на количество запросов за период времени. Поэтому
при неудачной проверке (если получен пустой ответ или запрос завершился ошибкой) выполняется
повторный запрос.

Модели также имеют различные интерфейсы взаимодействия. Для локальной модели применяется
один интерфейс взаимодействия, для удалённой, предоставляемой сторонним сервисом, — другой.

*Рисунок 3 — Схема отправки запросов к модели*

Если работа студента была проверена ранее и содержала выявленные недостатки, может быть
выполнена повторная проверка. При этом используется результат предыдущей проверки (если он
существует) и формируется запрос, содержащий предыдущую и новую версии отчёта. Модель
выполняет проверку и сравнение отчётов: если недочёты были исправлены, модель повышает
оценку; в противном случае модель указывает на сохраняющиеся недостатки и рекомендует
доработать отчёт.

Для повышения качества проверки предусмотрена возможность проверки с использованием
нескольких моделей. Один и тот же отчёт проверяется разными моделями, после чего заранее
выбранная модель формирует сводную рецензию.

Такой подход обладает рядом преимуществ. Различные модели нередко акцентируют внимание на
разных проблемах даже в рамках одного и того же отчёта; кроме того, результат проверки
зависит от уровня модели. Поскольку отчёт проверяется несколько раз разными моделями, итоговая
оценка оказывается более объективной и охватывает разные стороны работы студента.

### Структура программного комплекса

Серверная часть организована по модульному принципу: каждой предметной области соответствует
отдельный модуль. Ниже приведены основные классы-сервисы и их методы.

Класс `ProviderService` предназначен для управления сущностями провайдеров. Класс содержит
следующие методы:

- `findMany` — возвращает все записи провайдеров;
- `findOne` — возвращает одну запись провайдера по указанному идентификатору;
- `create` — создаёт новую запись провайдера на основе переданного объекта передачи данных
  (Data Transfer Object) и сохраняет её в базе данных;
- `update` — обновляет существующую запись провайдера;
- `delete` — удаляет запись провайдера.

Класс `KeyService` отвечает за операции с ключами доступа. Класс содержит следующие методы:

- `findOne` — возвращает ключ по идентификатору;
- `findMany` — возвращает все записи ключей;
- `create` — создаёт новую запись ключа;
- `update` — обновляет существующую запись ключа;
- `delete` — удаляет запись ключа.

Класс `ModelService` предоставляет функции управления сущностями моделей, используемых для
взаимодействия с различными интерфейсами больших языковых моделей. Класс содержит следующие
методы:

- `findByIds` — возвращает список моделей по заданному массиву идентификаторов;
- `findOne` — возвращает одну запись модели по идентификатору;
- `findMany` — возвращает все записи моделей, включая связанные данные о ключах и провайдерах;
- `create` — создаёт модель с учётом указанного интерфейса взаимодействия;
- `update` — обновляет модель с учётом указанного интерфейса взаимодействия;
- `delete` — удаляет запись модели из базы данных.

Класс `GroupService` управляет сущностями групп. Класс содержит следующие методы:

- `findOne` — возвращает одну запись группы по идентификатору;
- `findMany` — возвращает все записи групп;
- `create` — создаёт новую запись группы;
- `update` — обновляет существующую запись группы;
- `delete` — удаляет запись группы из базы данных;
- `isMember` — проверяет, входит ли студент в указанную группу;
- `addMember` — добавляет студента в группу;
- `removeMember` — удаляет студента из группы;
- `getGroupStudents` — возвращает список студентов, принадлежащих указанной группе.

Класс `StudentService` управляет операциями над сущностями студентов. Класс содержит
следующие методы:

- `findByIds` — возвращает список студентов по заданному массиву идентификаторов;
- `findRawStudent` — возвращает студента по полному совпадению имени, фамилии и отчества;
- `findOne` — возвращает одну запись студента по идентификатору;
- `findMany` — возвращает все записи студентов;
- `searchStudents` — выполняет поиск студентов с постраничным выводом и фильтрацией;
- `create` — создаёт новую запись студента;
- `update` — обновляет существующую запись студента;
- `delete` — удаляет запись студента из базы данных.

Класс `LabService` управляет сущностями лабораторных работ. Класс содержит следующие методы:

- `findOne` — возвращает одну запись лабораторной работы;
- `findMany` — возвращает все записи лабораторных работ;
- `prepareFile` — обрабатывает загруженный файл с заданием;
- `create` — создаёт новую запись лабораторной работы;
- `update` — обновляет существующую запись лабораторной работы;
- `delete` — удаляет запись лабораторной работы из базы данных.

Класс `CheckService` управляет операциями, связанными с результатами проверок. Класс содержит
следующие методы:

- `getStudentChecks` — возвращает все записи проверок студентов;
- `getByIds` — возвращает список проверок по заданному массиву идентификаторов;
- `findLastCheck` — возвращает последнюю проверку для указанного студента;
- `create` — создаёт новую запись проверки;
- `getLabChecks` — возвращает и группирует все проверки для конкретной лабораторной работы;
- `findByLabs` — возвращает все проверки, связанные с конкретной лабораторной работой;
- `findByStudent` — возвращает проверки по идентификаторам лабораторной работы и студента.

Класс `CourseService` управляет сущностями курсов. Класс содержит следующие методы:

- `findOne` — возвращает одну запись курса по идентификатору;
- `findMany` — выполняет поиск курсов с постраничным выводом и фильтрацией по названию;
- `create` — создаёт новую запись курса;
- `update` — обновляет существующую запись курса;
- `delete` — удаляет запись курса из базы данных;
- `getLabs` — возвращает список лабораторных работ курса.

Класс `FileService` предоставляет функции работы с файловой системой, включая запись файлов,
разбор содержимого архивов и извлечение текста из документов различных форматов. Класс
содержит следующие методы:

- `writeFile` — записывает строковое содержимое в файл по указанному пути;
- `parseStudentsFromFile` — обрабатывает ZIP-архив, извлекая из него сведения о студентах;
- `parseArchive` — распаковывает ZIP-архив во временную директорию и вызывает `processFolders`
  для дальнейшей обработки содержимого;
- `processFolders` — обрабатывает содержимое корневой директории: сканирует поддиректории и
  для каждой из них, если имя папки соответствует ожидаемому формату, извлекает сведения о
  студенте и содержимое файлов внутри папки;
- `parseFolderName` — выполняет разбор имени папки для извлечения фамилии, имени, отчества и
  номера;
- `extractFolderContent` — извлекает текстовое содержимое из первого найденного файла формата
  PDF, DOCX или DOC в указанной папке;
- `parseFile` — извлекает текстовое содержимое из буфера файла в зависимости от его расширения
  (поддерживаются форматы PDF и DOCX; файлы иных типов читаются как текст в кодировке UTF-8);
- `extractPDF` — извлекает текст из PDF-файла, представленного в виде буфера;
- `extractDOCX` — извлекает текст из DOCX-файла, представленного в виде буфера.

Класс `ReportsService` управляет процессами проверки отчётов студентов с применением больших
языковых моделей. Класс содержит следующие методы:

- `parseStudentsFile` — выполняет разбор архива с отчётами студентов;
- `getLabChecks` — возвращает все проверки, связанные с указанной лабораторной работой;
- `getChecks` — возвращает записи проверок по списку их идентификаторов;
- `handleCheckReports` — определяет способ проверки отчётов (одной моделью или несколькими) и
  запускает её.

### Работа с большими языковыми моделями в программном комплексе

Провайдер представляет собой сущность, соответствующую удалённому сервису, который
предоставляет доступ к большим языковым моделям. Каждый провайдер обладает собственными
параметрами подключения, а именно названием и URL-адресом.

Доступ к большинству удалённых сервисов больших языковых моделей защищён ключами. Ключ
подтверждает, что запрос пользователя имеет право на использование ресурсов провайдера. Ключ
доступа применяется для удалённых провайдеров, поскольку именно при обращении к удалённому
сервису требуется подтверждение прав.

За управление ключами отвечает сервис `KeyService`. Он хранит ключи и обеспечивает их
привязку к конкретным моделям и провайдерам. Без корректно настроенного ключа удалённые
модели не смогут функционировать, поэтому при создании ключей необходимо убедиться в их
работоспособности.

Каждая модель обладает рядом характеристик, среди которых её название, системное имя, а также
параметры температуры и top_p. Эти параметры влияют на степень «креативности» либо, напротив,
предсказуемости ответа модели: чем выше температура, тем более разнообразными могут быть
ответы.

Каждая модель имеет собственный интерфейс взаимодействия `llmInterface`, определяющий способ
обращения к ней. Все операции, связанные с моделями, сосредоточены в сервисе `ModelService`.
Поддерживаются два интерфейса взаимодействия — `OpenAi` и `Ollama`.

Интерфейс, совместимый с `OpenAI`, применяется для моделей, доступных через удалённые сервисы
и требующих ключей доступа и провайдеров. Примерами могут служить модели самой компании
OpenAI, а также иные модели, предоставляющие совместимый прикладной программный интерфейс.
Для моделей с таким интерфейсом обязательно должны быть указаны и ключ доступа, и провайдер,
поскольку без них взаимодействие с моделью невозможно.

Интерфейс `Ollama` предназначен для взаимодействия с моделями, развёрнутыми локально, то есть
работающими на устройстве пользователя через платформу Ollama. Отличительной особенностью
таких моделей является то, что им не требуются отдельные ключи доступа или сторонние провайдеры.
Для их настройки достаточно указать базовые параметры модели.

Сервис `ModelService` учитывает указанные различия. При создании или обновлении модели
проверяется заданный интерфейс. Для модели, совместимой с `OpenAI`, проверяется наличие как
ключа, так и провайдера. Для модели с интерфейсом `Ollama` любая попытка привязать ключ или
провайдера отклоняется во избежание ошибок.

Основным сервисом, отвечающим за взаимодействие с большими языковыми моделями, является
`LlmService`. Его главная задача — управление процессом получения ответов и их обработка.

Сервис `LlmService` взаимодействует с конкретными провайдерами через специализированные
обработчики, реализующие интерфейс `ILlmProviderHandler`. Для создания нужного обработчика
применяется фабрика `LlmProviderFactory`. В зависимости от типа интерфейса фабрика
предоставляет соответствующий обработчик: `OpenAiHandler` — для взаимодействия с прикладным
программным интерфейсом OpenAI и `OllamaHandler` — для взаимодействия с локально развёрнутыми
моделями через Ollama.

Обработчик `OpenAiHandler` формирует запросы к сервису, используя базовый URL-адрес провайдера
и ключ доступа, привязанные к конкретной модели. Он также обрабатывает специфичные ошибки,
преобразуя ошибку прикладного программного интерфейса со статусом 400 в стандартную ошибку.

Обработчик `OllamaHandler` использует библиотеку Ollama для взаимодействия с локальными
моделями и реализует собственную логику обработки ошибок, специфичных для этого провайдера.

Метод `query` сервиса `LlmService` инкапсулирует логику повторных попыток и задержек. Он
получает нужный обработчик через `LlmProviderFactory` и вызывает его метод `completion`. Если
вызов завершается ошибкой или возвращает пустой ответ, сервис не прекращает работу немедленно,
а выполняет следующие действия:

- вызывает метод `processError` соответствующего обработчика для обработки ошибки;
- регистрирует предупреждение в журнале;
- выдерживает заданную задержку перед повторным выполнением запроса;
- повторяет запрос.

Описанный процесс повторяется до тех пор, пока не будет достигнуто предельное число попыток
`maxRetries`, заданное в настройках модели. Длительности пауз определяются параметрами
`queryDelay` (задержка перед запросом) и `errorDelay` (задержка после ошибки).

После успешного получения непустого ответа `LlmService` извлекает из него данные методом
`extractData`. Этот метод выполняет поиск в текстовом ответе специального маркера — блока,
заключённого в теги `<JSON>` и `</JSON>`.

Если блок найден, он извлекается, разбирается и проходит проверку на соответствие заранее
определённому формату объекта передачи данных средствами библиотек `class-transformer` и
`class-validator`. Если блок отсутствует, сформирован некорректно или не проходит проверку,
метод `extractData` выдаёт ошибку. Для повышения устойчивости разбора предусмотрены
дополнительные проходы: удаление обрамляющих ограничителей кода, экранирование управляющих
символов внутри строк и удаление завершающих запятых.

Когда преподаватель начинает проверку, он загружает ZIP-архив, содержащий отчёты студентов.
Приложение принимает архив и параметры проверки. К этим параметрам относятся идентификатор
лабораторной работы, выбранные модели (одна или несколько), список конкретных студентов для
проверки, а также признак, указывающий, следует ли учитывать предыдущие проверки студента.

Метод `handleCheckReports` выполняет проверку в фоновом режиме посредством вызова функции
`setImmediate`. Перед началом основной работы отправляется уведомление о запуске процесса
проверки. Если в ходе процесса возникает непредвиденная ошибка, система отправляет уведомление
о том, что отчёты студентов проверить не удалось. Следует отметить, что ответ на HTTP-запрос
возвращается до начала собственно проверки, поэтому об ошибках, возникающих в ходе проверки,
пользователь узнаёт через механизм уведомлений.

Сервис `FileService` выполняет разбор загруженного ZIP-архива. Предполагается, что внутри
архива каждый отчёт студента находится в отдельной папке, имя которой содержит сведения о
студенте (фамилию, имя, отчество). Эти сведения извлекаются из имён папок.

Далее из каждой папки извлекается файл отчёта. Сервис `FileService` читает содержимое таких
файлов, преобразуя его в текст. Для файлов формата PDF применяется библиотека `pdf-parse`, для
файлов формата DOCX — библиотека `mammoth`. Файлы иных форматов читаются как текст в кодировке
UTF-8.

Используя извлечённые из имён папок фамилию, имя и отчество, сервис `StudentService` выполняет
поиск студента в базе данных. Если студент не найден, для него создаётся новая запись.

Перед обращением к модели формируется запрос. Сервис `PromptService` строит запрос на основе
содержимого отчёта студента, текста задания лабораторной работы и текста промпта, привязанного
к курсу. Если установлен признак учёта предыдущих проверок и для студента существует предыдущая
оценка по данной работе, сервис `PromptService` дополняет запрос сведениями о предыдущей
проверке.

Сформированный запрос вместе со сведениями о выбранной модели передаётся в сервис `LlmService`,
который отвечает за непосредственное взаимодействие с моделью и поддерживает оба интерфейса
(`OpenAi` и `Ollama`). В зависимости от модели сервис направляет запрос к соответствующему
прикладному программному интерфейсу.

Каждый ответ модели записывается в журнальный файл; в журнал также выводятся сообщения об
ошибках, успешно выполненных запросах и предупреждениях.

После получения ответа, представляющего собой текстовую строку, сервис `LlmService` извлекает
из него данные. Запрос предписывает модели возвращать результат в формате JSON, заключённом в
теги `<JSON>` и `</JSON>`. Сервис находит этот блок, разбирает его и преобразует в объект со
следующими полями:

- оценка по десятибалльной шкале;
- краткий отзыв на работу;
- положительные стороны работы;
- недостатки работы.

Помимо перечисленных полей результат содержит сведения анализа безопасности, описанные в
разделе [«Защита от внедрения вредоносных инструкций»](#защита-от-внедрения-вредоносных-инструкций).

На каждом этапе проверки одного отчёта приложение отправляет пользователю сведения о ходе
проверки через сервис `NotificationService`. Этот сервис содержит следующие методы отправки
уведомлений:

- `reportOneStarted` — уведомляет о начале обработки одного отчёта;
- `reportOneChecked` — уведомляет об успешной проверке одного отчёта;
- `reportOneFailed` — уведомляет о неудачной обработке одного отчёта;
- `reportsChecked` — уведомляет о завершении проверки нескольких отчётов;
- `checkStarted` — уведомляет о начале процесса проверки отчётов;
- `checkFailed` — уведомляет о неудаче общего процесса проверки.

Метод `handleCheckReports` сервиса `ReportsService` получает объект передачи данных со всеми
необходимыми сведениями и выполняет проверку. Сначала он обращается к сервису `ReportStrategy`
для получения конкретной стратегии проверки в зависимости от числа переданных моделей. Затем
посредством `setImmediate` асинхронно запускается метод `check` выбранной стратегии. Внутри
этой асинхронной функции выполняется управление уведомлениями через `NotificationService` —
сообщения о начале проверки, её успешном завершении или о возникшей ошибке. Кроме того,
`ReportsService` предоставляет вспомогательные методы `parseStudentsFile` (разбор файла со
студентами), а также `getLabChecks` и `getChecks` (получение сведений о ранее выполненных
проверках), делегируя работу сервисам `FileService` и `CheckService` соответственно.

Сервис `ReportStrategy` реализует порождающий шаблон «Фабричный метод» для выбора способа
проверки. Его единственный открытый метод `getStrategy` принимает число — количество моделей.
Если это число равно двум или превышает его, метод, используя `ModuleRef` для разрешения
зависимостей, возвращает экземпляр `MultipleModelStrategy`; в противном случае возвращается
`OneModelStrategy`.

Стратегия `OneModelStrategy` предназначена для сценария с одной моделью. Её метод
`prepareCheckData` загружает все необходимые данные: лабораторную работу, модель, задание и
отчёты. Затем метод `check` формирует массив промисов, каждый из которых представляет собой
вызов `reportCheck.checkOneReport` для одного отчёта. Промисы выполняются параллельно
посредством `Promise.allSettled`. Результаты обрабатываются методом
`reportCheck.filterSuccessResults`, после чего успешные проверки сохраняются методом
`reportCheck.createChecks`.

Стратегия `MultipleModelStrategy` реализует проверку с использованием нескольких моделей.
Аггрегирующей назначается последняя модель из переданного списка; остальные модели выполняют
индивидуальные проверки. Метод `prepareCheckData` загружает данные лабораторной работы,
проверяющих моделей, аггрегирующей модели и отчёты. Затем метод `check` организует первый
этап: для каждой проверяющей модели формируется массив промисов `reportCheck.checkOneReport`
по всем отчётам, и эти массивы выполняются параллельно. После завершения первого этапа метод
`prepareMultipleData` собирает все успешные результаты и группирует их по студентам, формируя
для каждого студента набор его проверок разными моделями. На втором этапе метод
`combineCheckResults` обходит сгруппированные данные и для каждого студента вызывает
`combineCheckResult`. Последний формирует сводный запрос методом `promptService.prepareMultiplePrompt`,
получает итоговый ответ аггрегирующей модели, извлекает из него данные и формирует итоговый
результат. Наконец, итоговые результаты сохраняются в базе данных методом
`reportCheck.createChecks`.

### Защита от внедрения вредоносных инструкций

Поскольку отчёт студента является недоверенными данными, он может содержать инструкции,
направленные на манипулирование поведением модели (например, требование выставить максимальную
оценку, пропустить проверку или скрыть ошибки). Для противодействия подобным воздействиям в
программном комплексе предусмотрен отдельный модуль безопасности, центральным компонентом
которого является сервис `PromptInjectionService`.

Защита реализована в несколько рубежей.

Первый рубеж — детерминированный анализ. Метод `analyze` сервиса `PromptInjectionService`
применяет к тексту отчёта набор регулярных выражений, выявляющих типовые приёмы внедрения
инструкций: попытки изменить роль модели, потребовать завышенную оценку, пропустить проверку,
скрыть ошибки или нарушить формат вывода. По результатам анализа формируется заключение,
включающее признак обнаружения, уровень риска (`none`, `low`, `medium` или `high`), перечень
сработавших индикаторов и подозрительные фрагменты текста.

Второй рубеж — построение запроса. Сервис `PromptService` помещает текст отчёта и иные
недоверенные данные в специальные блоки, ограниченные тегами вида `<UNTRUSTED_*>`, и
обезвреживает внутри этих блоков служебные маркеры (`@SPLIT`, `<JSON>` и одноимённые теги).
Кроме того, шаблон запроса разделяется по маркеру `@SPLIT` на системную и пользовательскую
части, благодаря чему недоверенное содержимое отчёта попадает только в пользовательскую часть
запроса. Результат детерминированного анализа подставляется в шаблон в качестве контекста
безопасности, а сам шаблон содержит явные правила безопасности, предписывающие модели
рассматривать содержимое отчёта как данные, а не как инструкции.

Третий рубеж — обработка ответа модели. После извлечения результата метод `mergeResultFields`
объединяет сведения о безопасности, сообщённые моделью, с результатами детерминированного
анализа, выбирая более высокий уровень риска и устраняя повторяющиеся фрагменты. Затем метод
`assertGeneratedReviewAllowed` проверяет текст сформированной рецензии на наличие признаков
того, что модель последовала внедрённой инструкции (например, характерных фраз о пропуске
проверки по просьбе студента); при обнаружении таких признаков проверка данного отчёта
завершается ошибкой.

Объект передачи данных результата проверки `CheckResultDto` содержит соответствующие поля:
признак обнаружения внедрения, уровень риска, перечень подозрительных фрагментов и комментарий
безопасности. Эти поля сохраняются в таблице проверок и возвращаются клиенту; при обнаружении
попытки внедрения интерфейс отображает соответствующее предупреждение.

Следует учитывать, что детерминированный анализ на основе регулярных выражений охватывает
распространённые, но не все возможные приёмы. Поэтому правила безопасности в составе запроса
и проверка сформированной рецензии служат дополнительными рубежами защиты, и полагаться
исключительно на детерминированный анализ недопустимо.

## Руководство системного программиста

### Общие сведения о программе

Программный комплекс предназначен для рецензирования отчётов по лабораторным работам студентов.

Функции программного комплекса:

- проверка отчётов;
- настройка ключей, моделей и провайдеров;
- навигация по истории проверок;
- создание курсов, групп и студентов;
- выполнение проверок несколькими моделями;
- создание и редактирование промптов.

Для корректной работы приложения необходима следующая конфигурация технических средств:

- процессор с тактовой частотой 2,2 ГГц или выше, имеющий два или более ядра;
- объём оперативной памяти более 4 ГБ;
- наличие клавиатуры и мыши;
- операционная система Windows 10 или Windows 11;
- наличие подключения к сети Интернет.

### Структура программы

Приложение состоит из двух проектов. Первый проект — клиентское приложение на библиотеке
React, размещённое в каталоге `client`. Второй проект — серверное приложение на платформе
Node.js с использованием фреймворка NestJS, размещённое в каталоге `server`.

### Настройка программы

Для подготовки программного комплекса к работе необходимо:

- установить платформу Node.js;
- запустить файл `install.bat` в корневом каталоге проекта;
- дождаться завершения загрузки всех пакетов;
- по завершении установки запустить файл `start.bat`.

Серверной части требуется переменная окружения `DATABASE` — путь к файлу базы данных SQLite.
При её отсутствии сервер завершает работу при запуске.

В случае успешного выполнения запроса сервер возвращает ответ с результатом, в случае ошибки —
ответ с сообщением об ошибке. Методы тестирования программы приведены в таблице 10.
